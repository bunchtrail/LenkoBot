# LenkoBot Engineering Guide

## Источники истины

- Целевая архитектура MVP описана в `docs/architecture/mvp-spec.md`.
- Сохранённые исследования Hermes находятся в `docs/analysis/` и не должны дублироваться без новой причины или новой версии upstream.
- При изменении границы модулей, модели данных, внешнего контракта, security policy или пользовательского workflow сначала обновить архитектурную документацию в том же изменении.
- Если код противоречит архитектурной спецификации, не исправлять расхождение молча: подтвердить решение, обновить документацию и только затем продолжать реализацию.

## Исследование upstream

- Сначала использовать сохранённые исследования в `docs/analysis/`; не искать повторно в scratch/upstream checkout без конкретного пробела в документации.
- Если требуемого факта нет в `docs/analysis/`, сначала зафиксировать этот пробел и запускать внешний поиск только через subagent с профилем `gpt-5.6-terra`, thoroughness `medium`.
- Результат нового исследования сохранять в `docs/analysis/` с источником, версией и границами применимости; не переносить upstream-код без отдельного решения о provenance и лицензии.

## TDD

- Каждое изменение поведения начинается с теста, который воспроизводит требование или дефект и сначала падает.
- Рабочий цикл: `red -> green -> refactor`: минимальный тест, минимальная реализация, затем безопасное упрощение кода.
- Тестировать наблюдаемое поведение и инварианты, а не внутренние детали реализации.
- Исправление дефекта всегда сопровождается регрессионным тестом.
- Задача не завершена, пока не выполнены релевантные тесты и не проверены сценарии, затронутые изменением.

## Архитектура и документация

- Для существенных работ вести `docs/architecture/implementation-notes.md` с разделами: подтверждённые решения, находки, отклонения, оставшиеся неизвестности и проверка.
- Явно помечать решения как `Confirmed`, `Assumed` или `Open`.
- Документировать причины важных ограничений и инвариантов, а не журнал повседневных действий.
- Перед реализацией новой крупной возможности проводить короткий проход `finding-unknowns`: проверить код и документацию, ранжировать неизвестности по влиянию и спросить пользователя только о решениях, которые нельзя вывести из evidence.

## Принципы проектирования

- Применять KISS: выбирать наименьшее корректное решение, понятное без скрытых соглашений.
- Разделять код по предметной ответственности. У модуля должен быть один ясный владелец данных и поведения.
- Держать зависимости направленными и явными; не связывать модули через глобальное изменяемое состояние, неявные filesystem conventions или копирование логики.
- Создавать интерфейс или extension point только для подтверждённой вариативности. Не строить абстракции на будущее без конкретного сценария расширения.
- Новые возможности должны подключаться к узким контрактам существующих модулей, а не обходить их специальными ветками.
- Предпочитать небольшие композиционные модули большим универсальным классам. Не смешивать transport, domain logic, persistence и presentation в одном слое.
- Рефакторинг допустим только вместе с тестами, сохраняющими поведение; не расширять scope ради косметической перестройки.

## Рабочий порядок

1. Прочитать относящиеся к задаче разделы `docs/architecture/` и `docs/analysis/`.
2. Найти и зафиксировать высокоэффектные неизвестности; принять консервативные обратимые решения для остального.
3. Обновить архитектурный документ, если меняется долгоживущий контракт или инвариант.
4. Написать падающий тест, реализовать минимальное изменение и выполнить рефакторинг только после зелёного теста.
5. Запустить релевантную проверку, записать существенные отклонения и сообщить оставшиеся риски.

## Контроль версий

- Проект ведётся в Git; каждая завершённая feature-вертикаль оформляется отдельным осмысленным commit вместе с её тестами и архитектурной документацией.
- После прохождения проверок завершённый feature commit отправляется в настроенный GitHub remote до начала следующей feature-вертикали.
- Перед commit обязательно проверить `git status`, `git diff`, `git diff --check` и последние commits; stage только файлы текущей фичи.
- Не смешивать в одном commit несвязанные изменения, generated artifacts, локальные secrets, токены и credential stores.
- Commit message должен кратко описывать одну feature-вертикаль; amend, force-push и destructive git commands запрещены без явного запроса пользователя.
- Если репозиторий, remote или credentials не настроены, остановить version-control шаг, сообщить точную причину и запросить только недостающую настройку; не создавать выдуманный remote.
